## 1. ВИДЕНИЕ (Vision)

> **Зачем этот раздел:** Задаёт стратегическую рамку для всех решений — от архитектуры до микрокопирайтинга. Важен **всей команде**, особенно PO (Алексей) и PM при приоритизации, а также UX (Максим) при проектировании тона и сценариев.

### 1.1. Миссия

R-акселератор существует, чтобы **превратить хаос предпринимательской идеи в структурированный, инвесторски-готовый MVP за 2 дня** — вместо 6 месяцев в традиционном акселераторе.

Мы убираем три главных барьера начинающего фаундера:
1. **Некомпетентность** — не знает, что делать и в каком порядке.
2. **Одиночество** — нет команды экспертов и менторов.
3. **Стоимость** — акселераторы, консультанты, MBA-программы недоступны.

### 1.2. Ключевая ценность

**AI-команда из 5–6 узкоспециализированных экспертов**, доступная 24/7 по цене подписки. Каждый эксперт ведёт пользователя через свой этап: от анализа рынка до готового питчдека. На выходе — пакет инвесторских документов, прошедших внутренний скоринг качества.

### 1.3. Позиционирование

| Параметр | Традиционный акселератор | Конкуренты (ChatGPT, Notion AI) | R-акселератор |
|----------|--------------------------|----------------------------------|---------------|
| Время до результата | 3–6 месяцев | Не определено (инструмент без методологии) | 2 дня |
| Методология | Есть, но ручная | Отсутствует | Встроена в агентов |
| Стоимость | $10K–$50K+ или equity | $20–200/мес (без отраслевой экспертизы) | Подписка + токены |
| Персонализация | Высокая (менторы) | Низкая (generic) | Высокая (контекст проекта + профиль) |
| Масштабируемость | Ограничена менторами | Безлимитна, но без глубины | Безлимитна с сохранением глубины |
| Выходные артефакты | Разрозненные | Нет стандарта | Инвест-пакет, скоринг, API для партнёров |

### 1.4. Философия продукта

- **Интерфейс как у Perplexity:** чистый чат, боковое меню с проектами, глубокий контекст, мгновенный доступ к файлам. Минимум визуального шума — максимум продуктивности.
- **AI-first, human-validated:** Генерации ИИ — основа, но пользователь всегда контролирует и валидирует каждый артефакт (HITL-протокол).
- **Прогрессивное раскрытие:** Новичок получает простой чат-формат; опытный пользователь — Canvas, юнит-экономику, расширенный скоринг.
- **Платформенная амбиция:** От SaaS-инструмента к экосистеме, где стартапы встречаются с экспертами, менторами и инвесторами (маркетплейс + API для партнёров). Сетевые эффекты — ключевой стратегический актив.

### 1.5. AI-driven драйвер

R-акселератор — не «ещё один чат-бот». Это **оркестрированная система агентов**, где:
- Каждый ИИ-эксперт — связка нескольких узких агентов с чёткими входными/выходными чек-листами.
- R-менеджер — антропоморфный помощник аккаунта, не смешивающий задачи поддержки с задачами проекта.
- Контролёр-агент — самообучающийся модуль внутренней оценки качества генераций.
- Вся архитектура управляется State Machine с явными правилами переключения состояний (см. State Machine Logic.md).

### 1.6. Сетевые эффекты

```
Больше стартапов → привлекательнее для экспертов/партнёров
Больше экспертов → ценнее для стартапов
Больше данных → точнее скоринг → ценнее для инвесторов
Ценнее для инвесторов → привлекательнее для стартапов
```

Два типа эффектов:
- **Односторонние:** пользователи приглашают таких же пользователей (реферальная петля).
- **Перекрёстные:** стартапы привлекают экспертов, эксперты привлекают стартапы, данные привлекают инвесторов.

#### Открытые вопросы

- [ ] Определить KPI для измерения силы сетевых эффектов на каждом этапе (до маркетплейса, после маркетплейса).
- [ ] Уточнить порог критической массы стартапов, при котором запуск маркетплейса экспертов становится оправданным.
- [ ] Согласовать формулировку миссии для публичных коммуникаций (лендинг, PR).

### 4.10. Система метрик и аналитики

> **Зачем этот раздел:** Определяет, что именно измеряет продукт, как собираются данные и какие управленческие выводы они дают. Важен: **PO (Алексей)** — для принятия продуктовых решений, **PM** — для контроля здоровья продукта, **Backend (Иван)** — для проектирования системы событий, **LLM-эксперт (Павел)** — для оценки качества агентов.

#### Философия метрик — 4 уровня:
1. **Usage** — пользуются ли вообще
2. **Value** — получает ли результат
3. **Satisfaction** — доволен ли
4. **Monetization** — платит ли

#### Важное замечание:
> Telegram даёт только базовые метрики. Для оценки всех перечисленных метрик необходимо подключить **Google Analytics / Amplitude** или аналоги. Критически важно увязать Telegram ID пользователей с событиями на самом раннем этапе настройки аналитики.

## 5. R-TEAM: АРХИТЕКТУРА ИИ-ЭКСПЕРТОВ

> **Зачем этот раздел:** Описывает конвейер из 6 ИИ-экспертов R-Team — их роли, алгоритмы работы, форматы вывода, чек-листы и взаимодействие между собой. Критически важен для: **LLM-эксперта (Павел)** — как основной технический контракт на промпты; **Backend (Иван)** — для модели данных, хранения артефактов и pipeline-логики; **UX (Максим)** — для понимания пользовательского опыта внутри каждого эксперта; **PM** и **PO** — для понимания методологической ценности продукта.

### 5.1. Pipeline (конвейер экспертов)

```
Роман (R1) → Регина (R2) ⇄ Роза (R+) → Римма (R3) → Роберт (R4) → Рон (R5)
  Рынок       ЦА/JTBD    Синтетика    Продукт/Модель   MVP/Тесты     Питч
```

**Общая логика:** Пользователь проходит экспертов последовательно. Каждый эксперт принимает чёткий вход от предыдущего и отдаёт структурированный выход следующему. Нелинейность управляется явными обратными петлями.

#### Обратные петли

| Петля | Направление | Зачем | Пример |
|-------|-------------|-------|--------|
| **R2 ⇄ R+** | Регина отправляет к Розе → Роза возвращает данные Регине | Когда нет доступа к живым респондентам — Роза симулирует целевую аудиторию, Регина интерпретирует результаты | Регина предлагает 4 портрета ЦА и план интервью → пользователь не может провести реальные интервью → идёт к Розе → Роза симулирует 3 интервью → данные возвращаются Регине для финального вывода |
| **R4 → R1/R2/R3** | Роберт может вернуть на любой предыдущий этап | Если результаты экспериментов MVP показывают необходимость pivot — возвращаемся на нужный уровень | Тест лендинга показал: целевая аудитория не та → R4 рекомендует вернуться к R2 для пересмотра сегментов |
| **R5 → R4** | Рон может вернуть к Роберту | Если в процессе упаковки питча обнаруживается недостаток трекшна/метрик — усиливаем план экспериментов | Инвестор спрашивает о трекшне, а данных MVP недостаточно → R5 рекомендует ещё один цикл R4 |

| **Формат вывода для пользователя** | Структура питча: Интро → Проблема → Решение → Почему сейчас → Рынок → Трекшн → Конкуренты → Бизнес-модель → Команда → Инвестиционный запрос; плюс: Факты и метрики / Сильные стороны и точки роста / Рекомендации / План действий |
| **Внутренний чек-лист качества** | ✓ Все 10 блоков питча заполнены / ✓ Нарратив адаптирован под тип инвестора / ✓ Инвестиционный запрос конкретен (сумма, equity, use of funds) / ✓ Слабые места выявлены и проработаны |
| **Передача эстафеты** | Финальный эксперт. Может вернуть к R4 для усиления трекшна если данных недостаточно |

### 5.8. Общие принципы экспертов

> **Зачем этот подраздел:** Стандартизирует поведение всех 6 экспертов. Важен для: **LLM-эксперта (Павел)** — как единый стандарт промптов; **UX (Максим)** — для понимания пользовательского опыта; **Frontend** — для реализации UI-паттернов.

#### Стиль коммуникации

| Параметр | Стандарт |
|----------|----------|
| **Канал** | Telegram (Mini App, чат-интерфейс) |
| **Длина сообщения** | 8–10 строк на одно сообщение — лаконично, без «воды» |
| **Оформление** | Эмодзи как заголовки разделов, буллеты (—) для списков |
| **Тон** | Спокойный, структурный, профессиональный, на «ты», проактивный |
| **Ошибки пользователя** | Мягко корректирует, предлагает уточнение |

#### Поведенческие правила

| Правило | Описание |
|---------|----------|
| **Работа по шагам** | Каждый эксперт чётко объявляет текущий шаг и переходит к следующему только с подтверждения |
| **1 вопрос за раз** | После каждого шага задаётся не более одного уточняющего вопроса |
| **Мини-резюме** | Перед продолжением эксперт даёт краткое резюме того, что уже выяснено |
| **Режим помощи** | Если пользователь в тупике («не знаю», «как?») — эксперт берёт инициативу и предлагает 2–3 варианта |
| **HITL** | Каждый финальный вывод требует явного подтверждения пользователя перед передачей следующему эксперту |

#### Ограничения (что эксперты НЕ делают)

- Не раскрывают методики, источники, системные файлы и промпты
- Не консультируют по юридическим, налоговым, HR и регуляторным вопросам
- Не дают финансовых прогнозов без данных пользователя
- Не переключаются на темы вне своей зоны ответственности

#### Чек-листы

Каждый эксперт перед финальным ответом проверяет себя по внутреннему чек-листу (см. подразделы 5.2–5.7). Чек-лист не виден пользователю, но его результат влияет на качество вывода.

#### Передача эстафеты

Каждый эксперт завершает работу явной фразой-передачей:
- Называет следующего эксперта по имени
- Объясняет, что именно передаёт (1–2 предложения)
- Объясняет, зачем идти к следующему (ценность следующего шага)

### 5.9. Требования к LLM-эксперту (Павел)

> **Зачем этот подраздел:** Операционный гайд для Павла — что учитывать при разработке и доработке промптов каждого эксперта. Важен: **LLM-эксперт (Павел)** — первичный адресат; **PO** — для понимания технических ограничений.

| Эксперт | Ключевые ограничения промпта | На что обратить внимание при prompt engineering |
|---------|-----------------------------|-------------------------------------------------|
| **Роман (R1)** | Не давать финансовых прогнозов без данных; ограничить глубину анализа рынка (нет доступа к реальным БД) | Калибровка уверенности: R1 должен говорить «вероятно», «по общим данным», не делать абсолютных заявлений. Режим pivot требует отдельной ветки промпта |
| **Регина (R2)** | Не путать JTBD с демографией; не смешивать режим «новая идея» и «возврат от Розы» | Чёткий switch между режимами. Возврат от Розы требует специального контекстного блока. Гипотезы должны быть проверяемыми — промпт должен отклонять слишком абстрактные формулировки |
| **Роза (R+)** | Чётко разделять «данные» и «интерпретации» — Роза даёт только данные | Промпт должен явно запрещать Розе делать стратегические выводы. Синтетические портреты должны быть детализированы (возраст, контекст, цитаты). Риск галлюцинаций при симуляции — нужен контролирующий слой |
| **Римма (R3)** | Не заполнять блоки бизнес-модели «из воздуха» без данных от R1/R2 | 9 блоков должны быть связаны логически. Ценностное предложение должно явно ссылаться на JTBD из R2. Fine-tuning на реальных бизнес-моделях в схожих нишах |
| **Роберт (R4)** | Не рекомендовать MVP-форматы без привязки к гипотезам; не игнорировать ресурсные ограничения | Пороги успеха должны быть числовыми. Промпт должен требовать от R4 явного указания условий pivot. Сложный кейс: интерпретация противоречивых результатов тестов |
| **Рон (R5)** | Не фантазировать о трекшне; не давать нереалистичных оценок раунда | Нарратив питча зависит от типа инвестора — промпт должен учитывать этот параметр. Слабые места выявляются системно, не «по настроению». Инвестиционный запрос должен содержать конкретные цифры |

**Общие требования к промптам:**
- Каждый промпт версионируется в системе (см. 6.2.1)
- Тестирование в sandbox перед публикацией
- Rollback при деградации метрик качества (CSAT < 3.5)
- Отдельный промпт для каждого режима работы (не один «умный» промпт на все случаи)

#### Открытые вопросы для Павла

- [ ] Определить архитектуру передачи контекста между экспертами: full history, summary, или structured JSON?
- [ ] Согласовать базовую LLM-модель для каждого эксперта (GPT-4o, Claude 3.5 Sonnet, Gemini 1.5 Pro — в зависимости от задачи).
- [ ] Разработать систему детектирования «некорректного режима» (например, пользователь идёт к R3 без данных от R1/R2).
- [ ] Определить стратегию fine-tuning: на каких данных, когда начинать.

### 5.10. Требования к Backend (Иван)

> **Зачем этот подраздел:** Технический контракт для Backend на хранение и управление данными R-Team. Важен: **Backend (Иван)** — первичный адресат; **LLM-эксперт (Павел)** — для понимания архитектурных ограничений.

#### Модель данных для хранения промптов

```
Expert
  id (UUID)
  codename (R1, R2, R+, R3, R4, R5)
  name (Роман, Регина, ...)
  role (string)
  is_active (bool)
  current_prompt_version_id (FK)
  llm_model (string)
  temperature (float)
  max_tokens (int)
  created_at, updated_at

ExpertPromptVersion
  id (UUID)
  expert_id (FK)
  version_number (string: "1.1", "1.2", ...)
  prompt_text (text)
  mode (enum: new_idea, existing_product, pivot, return_from_rosa, ...)
  created_by (FK → AdminUser)
  created_at
  published_at (nullable)
  is_current (bool)
  sandbox_test_results (JSON)
  notes (text)
```

#### Версионирование промптов

| Требование | Реализация |
|-----------|-----------|
| История версий | Полная история всех версий промпта для каждого эксперта |
| Rollback | Возможность вернуться к любой предыдущей версии одним действием |
| Diff-просмотр | Сравнение двух версий промпта (как в Git) |
| Sandbox-тест | Тестирование версии на тестовых данных до публикации |
| Публикация | Явное действие «Опубликовать» с подтверждением; текущая версия не меняется до подтверждения |

#### Связи между экспертами (pipeline-логика)

```
ExpertPipelineEdge
  id (UUID)
  from_expert_id (FK)
  to_expert_id (FK)
  edge_type (enum: sequential, loop_forward, loop_back)
  trigger_condition (text: описание условия перехода)
  is_active (bool)

ExpertSession (диалог пользователя с экспертом)
  id (UUID)
  project_id (FK)
  expert_id (FK)
  mode (enum)
  status (enum: in_progress, completed, paused)
  started_at, completed_at
  context_payload (JSON: входные данные от предыдущего эксперта)
  output_payload (JSON: артефакты для следующего)
```

#### Хранение артефактов экспертов

| Эксперт | Тип артефакта | Формат хранения |
|---------|--------------|-----------------|
| R1 — Роман | Рыночный бриф | JSON (structured) + markdown для отображения |
| R2 — Регина | Сегменты ЦА + JTBD + гипотезы + план тестов | JSON (structured) |
| R+ — Роза | Синтетические портреты + данные симуляций | JSON (raw data, без интерпретации) |
| R3 — Римма | Value Proposition Canvas + 9 блоков бизнес-модели + гипотезы MVP | JSON (structured по блокам) |
| R4 — Роберт | Формат MVP + план экспериментов + метрики + карта обучения | JSON (structured) |
| R5 — Рон | 10 блоков питча + инвестиционный запрос | JSON → PDF-шаблон |

**Требования к хранилищу артефактов:**
- Каждый артефакт привязан к `ExpertSession` и `project_id`
- Артефакты immutable после финализации (только добавление новых версий)
- PDF-экспорт генерируется on-demand из JSON
- Черновики артефактов (до HITL-подтверждения) хранятся отдельно от финализированных

#### Открытые вопросы для Ивана

- [ ] Определить стратегию хранения большого JSON-контекста (S3 vs PostgreSQL jsonb vs отдельная таблица).
- [ ] Согласовать формат `context_payload` для передачи между экспертами — нужен ли единый schema contract.
- [ ] Определить политику retention для черновиков артефактов.
- [ ] Спроектировать индексы для быстрого поиска артефактов проекта по стадии/эксперту.

## 7. ТРЕБОВАНИЯ К СТРУКТУРНЫМ СХЕМАМ СТРАНИЦ

> **Зачем этот раздел:** Задаёт требования к wireframe'ам до начала их проектирования. Обеспечивает единое понимание между UX-дизайнером (Максим), UI-дизайнером (Катя), фронтенд-разработчиком (Максим в роли vibe-coder) и бэкенд-разработчиком (Иван). Содержит спецификации для каждого ключевого экрана.

### 7.0. Общие принципы

| Принцип | Описание |
|---------|----------|
| **Mobile-first** | Проектируем сначала для мобильных, затем масштабируем на десктоп. Мобильная версия — полноценный продукт (за исключением Canvas) |
| **Философия Perplexity** | Чистый чат в центре, боковое меню с проектами (collapsible), глубокий контекст, мгновенный доступ к файлам. Минимум визуального шума |
| **Кнопки где можно, текст где необходимо** | Предпочитаем кнопки и quick actions в чат-интерфейсе. Свободный текстовый ввод — для детализации, не для навигации |
| **Прогрессивное раскрытие** | Сложные функции появляются по мере продвижения. Новичок не видит то, что ему не нужно |
| **Омниканальность** | Web — основа. Архитектура готова к расширению на мессенджеры (Telegram, WhatsApp) в будущем. Диалоговый формат — универсальный |
| **Адаптивность** | Breakpoints: mobile (<768px), tablet (768–1024px), desktop (>1024px) |

| Термин | Определение | Контекст использования |

| **R-акселератор** | B2B SaaS-сервис для бизнес-моделирования, превращающий идею предпринимателя в структурированный MVP и инвесторский пакет документов с помощью ИИ-экспертов | Название продукта. Используется во всех внешних и внутренних коммуникациях |

| **R-Team** | Команда из 6 ИИ-экспертов: Роман (R1), Регина (R2), Роза (R+), Римма (R3), Роберт (R4), Рон (R5). Работают последовательно через pipeline с обратными петлями | Внутреннее название ИИ-команды продукта. См. раздел 5 |
| **Pipeline экспертов** | Конвейер R1→R2⇄R+→R3→R4→R5, по которому проходит пользователь с проектом. Управляется State Machine с обратными петлями | Архитектурный термин. Описан в разделе 5.1 |
| **Агент** | Узкоспециализированный ИИ-модуль, выполняющий конкретную задачу в рамках ИИ-эксперта. Не взаимодействует с пользователем напрямую | Внутренний термин архитектуры. Пользователь видит «Эксперта», не «Агента» |
| **Canvas** | Интерактивное визуальное полотно для структурирования информации. Атрибут ИИ-эксперта (~200 канвасов в системе). Доступен только в Premium на десктопе | Канвас — дополнение к чату, не замена. Для визуальных тонкостей бизнес-моделирования |
| **Этап** | Логический шаг в пути бизнес-моделирования, соответствующий одному ИИ-эксперту. Этапы проходятся последовательно (или с калибровкой) | Последовательность этапов определяется Expert Matrix.md |
| **Диалог** | Сессия чат-общения пользователя с конкретным ИИ-экспертом в рамках проекта. Каждый переход к новому эксперту создаёт новый диалог | Один проект содержит несколько диалогов (по числу экспертов). История диалогов сохраняется |
| **Онбординг** | Процесс первичной квалификации пользователя через чат с R-менеджером: определение роли, отрасли, стадии и ожиданий. Результат — автоматическое создание проекта | Сценарий описан в Onboarding CJM.md |
| **Артефакт** | Любой файл/документ, сгенерированный ИИ-экспертом или загруженный пользователем: меморандум, питчдек, таблица, summary и т.д. | Артефакты хранятся в проекте, привязаны к этапам. Экспортируются в PDF |
| **Токен** | Единица оплаты LLM-генераций. Покупаются в рамках подписки или отдельно. Сгорают в конце месяца. Обнуляются при смене контекста проекта | Не путать с «токенами» в контексте LLM (единицы текста). Здесь токен = единица биллинга |
| **Подписка** | Уровень доступа к функционалу: Базовый, Оптимальный, Премиум. Определяет лимиты проектов, тиммейтов, доступ к Canvas | Три тарифных плана, см. раздел «Монетизация» |
| **Summary** | Автоматически сгенерированная выжимка ключевой информации о проекте или этапе. Обновляется по мере прогресса | Используется для быстрого контекста (для менторов, для калибровки) |
| **Чек-лист вход/выход** | Набор обязательных данных, необходимых эксперту для начала работы (вход) и перечень артефактов, которые он должен создать (выход) | Определяет контракт между этапами. Описывается в Expert Matrix.md |
| **Скоринг** | Внутренняя оценка качества проекта и/или пользователя. Рассчитывается на основе big data логов, поведенческого профиля, семантики запросов, дисциплины использования | Данные обезличенные. Используются для внутренней аналитики и API для партнёров |
| **Инвесторский меморандум** | Структурированный документ, описывающий бизнес-модель, рынок, команду, финансовые прогнозы. Формат: краткий и подробный | Финальный артефакт проекта. Экспортируется в PDF, доступен через API для партнёров |
| **One-pager** | Одностраничное краткое описание проекта (мемо). Содержит суть идеи, рынок, модель монетизации, ключевые метрики | Быстрый формат для первичного знакомства инвестора/руководства с проектом |

| **Context Strategy** | Стратегия управления памятью и контекстом ИИ-агентов: какие данные передаются между экспертами, как сжимается контекст, как обновляется summary | Описана в Context Strategy.md. Критически важна для LLM-эксперта (Павел) |
| **Калибровка** | Процесс дополнения недостающих данных, когда пользователь приходит к эксперту не с первого этапа. Эксперт запрашивает ключевые данные предыдущих этапов | Позволяет стартаперам (стадия Seed+) не проходить все этапы с нуля |
| **Пространство** | Контейнер для управления проектами и доступами. Пространство = аккаунт пользователя + его проекты + приглашения | Аналог Workspace в Notion / Space в Perplexity |

| **Профиль проекта** | Набор контекстных данных проекта: роль пользователя, отрасль, стадия, цели, ограничения. Отделён от аккаунта пользователя | Используется для персонализации генераций. Заполняется при онбординге, редактируется в настройках проекта |

### В.1. КЛИЕНТСКИЕ СЕГМЕНТЫ

---

#### В.1.1. Индивидуальный фаундер — от идеи к питчдеку через R1→R2→R+→R3→R4→R5

**Персонаж:** Маша, 32 года, маркетолог в найме, идея — сервис подбора нянь по совместимости

**Шаг 1. Первый контакт**
Маша видит рекламу R-акселератора в Telegram, переходит по ссылке. Открывается Mini App. Она видит короткий экран приветствия с одной кнопкой «Начать».

**Шаг 2. Регистрация**
Вводит email → получает код на почту → вводит 6-значный код → аккаунт создан. Весь процесс занимает 45 секунд.

**Шаг 3. Онбординг с R-менеджером**
R-менеджер здоровается и задаёт вопросы кнопками:
- Роль: «Начинающий предприниматель»
- Отрасль: «Сервисы»
- Стадия: «Идея»
- Ожидание: «Хочу понять, стоит ли двигаться дальше»
R-менеджер резюмирует ответы и нажимает «Создать проект».

**Шаг 4. Создание проекта**
Система автоматически создаёт проект «Проект 1» с профилем. Маша видит рабочую область с боковым меню и центральным чатом. R-менеджер объясняет, что её первый шаг — Роман (R1), аналитик рынка.

**Шаг 5. Работа с Романом (R1) — анализ рынка**
Роман здоровается и просит описать идею. Маша пишет несколько предложений. Роман задаёт уточняющие вопросы по одному. Шаги: контекст → рынок → ниши → выбор. После 3–4 сообщений Маша получает рыночный бриф: 2 варианта ниши, ландшафт конкурентов, обоснование. Маша подтверждает (HITL). Роман передаёт эстафету Регине.

**Шаг 6. Работа с Региной (R2) — анализ аудитории**
Регина принимает бриф от Романа автоматически. Предлагает 3 сегмента ЦА. Маша выбирает «Работающие мамы». Регина углубляет анализ: JTBD, боли, предлагает 3 гипотезы. Маша видит, что не может провести живые интервью прямо сейчас. Регина предлагает обратиться к Розе.

**Шаг 7. Работа с Розой (R+) — синтетические интервью**
Роза создаёт 3 синтетических портрета «работающих мам» и симулирует интервью. Маша получает ответы на вопросы о болях и барьерах. Данные возвращаются Регине.

**Шаг 8. Возврат к Регине с данными**
