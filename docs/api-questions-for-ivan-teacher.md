# Контракт saas-teacher ↔ n8n (для Ивана)

Этот документ фиксирует, что saas-teacher отправляет в n8n webhook и в каком формате ожидает ответ. Реализация на нашей стороне — `apps/saas-teacher/src/lib/agents/n8n.ts`.

## Каждый агент — отдельный webhook

В админке CMS в карточке AI-агента есть поле `n8nWebhookUrl` и опциональный `n8nSecret`. Каждый агент маршрутизируется на свой workflow, OpenAI-инструкции и assistant_id хранятся внутри n8n.

## Запрос

`POST {agent.n8nWebhookUrl}`

Заголовки:

| Header | Назначение |
|---|---|
| `Content-Type: application/json` | |
| `Accept: text/event-stream, application/json` | предпочтительно стрим, fallback — обычный JSON |
| `X-Rocketmind-Signature: sha256=<hex>` | HMAC-SHA256(body, n8nSecret), присылается только если secret задан |

Тело:

```jsonc
{
  "agent": {
    "id": "uuid",
    "slug": "lesson-1-business-model",
    "name": "Бизнес-наставник",
    "role": "Бизнес-моделирование"
  },
  "message": "<текст сообщения пользователя>",
  "context": {
    "project": {
      "id": "uuid",
      "name": "Бизнес-модель для AI-консалтинга",
      "profile": { /* произвольный JSON из StudentProject.profile */ }
    },
    "student": {
      "id": "uuid",
      "email": "user@example.com",
      "firstName": "Алексей",
      "role": "Основатель"
    },
    "history": [
      { "role": "user",      "content": "..." },
      { "role": "assistant", "content": "..." }
    ],
    "artifacts": [
      { "kind": "value_prop",  "content": { /* json */ } },
      { "kind": "persona",     "content": { /* json */ } }
    ],
    "summaries": [
      { "agentSlug": "intro-agent", "summary": "Прошёл онбординг, выбрал нишу X..." }
    ]
  }
}
```

Замечания:
- `history` — переписка только с **этим** агентом по этому проекту (StudentSession.messages), последние 50 сообщений.
- `artifacts` — последние 20 артефактов по проекту, со всех агентов. Это «доска проекта».
- `summaries` — авто-сводки от **других** агентов этого же проекта. Так агенты «знают» что было до них, не глядя на сырые истории.
- Поле `profile` — произвольный JSON, который мы заполняем в онбординг-модалке. Сейчас там `null` или поля профиля; формат закрепим, когда определим набор вопросов.

## Ответ

### Вариант A — Server-Sent Events (предпочтительный)

`Content-Type: text/event-stream`

Каждое событие — стандартный SSE-чанк:

```
data: {"type":"delta","text":"Привет"}

data: {"type":"delta","text":", давай"}

data: {"type":"artifact","kind":"value_prop","title":"Ценностное предложение","content":{"who":"...","problem":"..."}}

data: {"type":"summary","summary":"Обсудили нишу и аудиторию, выделили 3 сегмента"}

data: {"type":"done"}

data: [DONE]
```

Поддерживаемые типы событий:

| `type` | Поля | Семантика |
|---|---|---|
| `delta` | `text: string` | очередной кусочек ответа, конкатенируется на клиенте |
| `artifact` | `kind: string`, `title?: string`, `content: any` | агент создал артефакт (канвас, документ). Сохраняем в `StudentArtifact`, чтобы другие агенты видели его в контексте |
| `summary` | `summary: string` | свежая авто-сводка диалога. Пишем в `StudentSession.summary` |
| `done` | — | стрим закончен. После этого можно прислать `[DONE]` |

Если стрим обрывается без `done` — на нашей стороне финал-сохранение всё равно произойдёт, но без summary/artifacts из этого ответа.

### Вариант B — обычный JSON

`Content-Type: application/json`

```jsonc
{
  "content": "Полный текст ответа агента",
  "summary": "...",       // опц.
  "artifacts": [          // опц.
    { "kind": "value_prop", "title": "...", "content": { /* */ } }
  ]
}
```

Этот формат проще на стороне n8n, но без стриминга — пользователь увидит ответ целиком после паузы. Используем для быстрых прототипов; production-агенты — через SSE.

## Безопасность

- Если у агента в админке задан `n8nSecret`, мы подписываем тело HMAC-SHA256 и кладём в `X-Rocketmind-Signature`. На стороне n8n до выполнения workflow проверь подпись.
- Тело подписи = ровно тот байтовый JSON, что мы посылаем (без переформатирования).

## Открытые вопросы Ивану

1. Какой URL-формат вебхуков ты предпочитаешь: один webhook на агента, или один master-webhook с роутингом по `agent.slug` внутри n8n? (Мы заложились на первый вариант — он гибче и не требует enum'а слагов в коде workflow.)
2. Поддерживает ли наш n8n потоковую отдачу SSE? Если нет — стартуем на варианте B (JSON), позже переключимся.
3. Где у тебя живут OpenAI ключи: per-workflow или один общий? Не критично для нас, но влияет на изоляцию ошибок.
4. Какие OpenAI-модели предполагаются по умолчанию (Assistants/Threads vs Chat Completions)? Хотим знать для рассчёта стоимости и таймаутов.
5. Тайм-аут на ответ — какой ставить на клиенте? Сейчас `fetch` без явного timeout; будем добавлять `AbortController` со значением, которое ты подтвердишь.
6. Артефакты: ты будешь возвращать их сам как `artifact` события, или нам парсить их из текста ответа? Мы рассчитываем на первый вариант.
7. Авто-сводки: как часто триггерить? Каждое сообщение, каждые N сообщений, по концу сессии? Сейчас мы сохраняем последний `summary` каждый раз, когда он приходит.
