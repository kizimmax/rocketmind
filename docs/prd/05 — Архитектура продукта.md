## Frontend

| Параметр | Значение |
|---|---|
| Framework | Next.js (App Router) |
| UI-библиотека | `@rocketmind/ui` (shadcn/ui + Tailwind CSS v4) |

| Стилизация | CSS variables, токены из DS |

### Ключевые модули фронтенда
- **Auth** — 2-step login (email + code)
- **Cases** — CRUD кейсов, сайдбар навигация
- **Chat** — real-time диалог с ИИ-агентом
- **Agents** — каталог + добавление в кейс
- **Bootstrap** — обработка URL `/a/{agent_slug}`

---

## Backend / Logic: n8n

n8n выступает как основной бэкенд для бизнес-логики:

- Приём сообщений пользователя через вебхуки
- Маршрутизация к нужному ИИ-агенту
- Управление контекстом диалога
- Триггеры оплаты (системные сообщения со ссылками)
- Генерация summary по завершении сценария

## Data Model

### Сущности

```
users
├── id (PK)
├── email
├── created_at
└── last_login

cases
├── id (PK)
├── user_id (FK → users)
├── name
├── status (active | archived)
├── created_at
└── updated_at

agents
├── id (PK)
├── slug (unique) — для URL /a/{agent_slug}
├── name
├── description
├── avatar_url
└── config (JSON — параметры для n8n)

case_agents (M2M)
├── case_id (FK → cases)
└── agent_id (FK → agents)

conversations
├── id (PK)
├── case_id (FK → cases)
├── agent_id (FK → agents)
├── created_at
└── updated_at

messages
├── id (PK)
├── conversation_id (FK → conversations)
├── role (user | assistant | system)
├── content (text)
├── metadata (JSON — ссылки на оплату, summary и пр.)
├── created_at
└── is_read
```

## Открытые решения

### ❓ DECISION-01: Персистентное хранилище

Нужно выбрать, где хранятся данные. Варианты:

| Вариант | Плюсы | Минусы |
|---|---|---|
| **Supabase (Postgres)** | SQL, Row-Level Security, встроенный Auth, Realtime подписки, edge functions | Зависимость от Supabase-инфраструктуры |
| **Firebase (Firestore)** | Уже используется в apps/internal, NoSQL гибкость, Firebase Auth | NoSQL для реляционных данных (cases→agents→messages) — неудобно, сложные запросы |
| **Postgres (self-hosted)** | Полный контроль, SQL для реляционной модели | Нужно поднимать/обслуживать, нет встроенного Auth |

**Рекомендация:** Supabase — реляционная модель хорошо ложится на Postgres, встроенный Auth покрывает 2-step login, Realtime нужен для чата.

**Статус:** ⏳ Требует решения

---

### ❓ DECISION-02: API-слой между фронтом и n8n

| Вариант | Плюсы | Минусы |
|---|---|---|
| **n8n напрямую** | Проще, меньше кода, быстрее MVP | Нет валидации на стороне фронта, CORS-зависимость, сложнее auth-проверки |
| **Next.js Route Handlers → n8n** | Валидация, auth-мидлвара, скрытие n8n URL, типизация | Дополнительный слой, дублирование |

**Рекомендация:** Next.js Route Handlers — даёт auth-проверку перед отправкой в n8n, скрывает внутренние URL, единая точка для логирования.

**Статус:** ⏳ Требует решения

## Ограничения MVP

- Нет WebSocket для чата (polling или Supabase Realtime)
- Нет очередей — n8n обрабатывает синхронно через вебхуки
- Нет кэширования — все запросы идут напрямую
- Нет CDN для пользовательского контента
